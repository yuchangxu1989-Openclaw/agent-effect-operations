## 1. 产品愿景与定位

AEO（Agent Effect Operations）是 Agent 效果运营平台，Self-Evolving Harness 的质量反馈基础设施。

Agent 执行任务的效果会随时间漂移——模型升级、知识库变化、Prompt 调整、用户行为变化都可能导致效果波动。AEO 自动度量 Agent 效果、发现漂移、诊断根因、生成优化方案，并驱动数据闭环验证优化是否生效。

AEO 解决的核心问题：Agent 效果好不好，目前靠人凭感觉判断。没有量化指标、没有自动告警、没有根因定位、没有闭环验证。效果变差了不知道，知道了不知道为什么，修了不知道有没有修好。AEO 把效果运营从人工感知升级为数据驱动的自动闭环。

设计硬约束：

- 效果度量必须可量化、可对比、可溯源，不接受主观评价作为唯一依据。
- 漂移检测必须自动触发，不依赖用户手动发起。
- 根因诊断必须给出可验证的归因，不能只说"可能是 X"。
- 优化建议必须可执行，不能只给方向不给方案。
- 闭环验证必须有基线对比，不能只看优化后的绝对值。
- 核心逻辑与宿主环境解耦，不写死对 OpenClaw 的依赖。

### 用户角色

核心用户是 Agent 运营者：独立产品操盘者、Agent 开发者、系统管理员。他们需要知道 Agent 干得好不好、哪里变差了、为什么变差、怎么改、改了有没有用。

### 关键使用场景

- 模型升级后，自动对比升级前后的效果差异，发现回归。
- Agent 日常运行中，持续监控关键指标，效果下降时自动告警。
- 收到漂移告警后，自动诊断根因并给出修复建议。
- 执行优化方案后，自动验证效果是否恢复或提升。
- 定期生成效果报告，展示趋势、问题和改进成果。

### 成功指标

- 漂移检测召回率 ≥ 90%（实际发生的效果漂移中被自动发现的比例）。度量方法：基于人工标注的漂移事件集，系统检测命中的事件占比。
- 根因诊断准确率 ≥ 70%（诊断结论经人工确认为正确的比例）。
- 优化建议采纳率 ≥ 60%（建议被执行的比例）。
- 闭环验证覆盖率 100%（每次优化执行后必须有验证结果）。

## 2. 目标用户画像

### 2.1 Solo Founder / 独立产品操盘者

- 用 Agent 推进产品研发、运营和决策。
- 需要知道 Agent 的表现是在变好还是变差，不想靠感觉判断。
- 需要在效果下降时收到告警和修复建议，不想自己排查。

### 2.2 Agent 开发者

- 构建和维护 Agent 系统，频繁调整 Prompt、模型、工具链。
- 需要每次调整后快速看到效果变化，判断改动是正向还是负向。
- 需要根因诊断帮助定位问题，减少盲目调试。

### 2.3 系统管理员

- 负责 Agent 集群的运行质量和 SLA。
- 需要全局效果仪表盘，一眼看到哪些 Agent 状态异常。
- 需要效果趋势报告，支撑容量规划和资源调配决策。

## 3. 功能需求（FR）

### 域 A：效果度量（Effect Measurement）

定义和采集 Agent 效果指标，建立量化评估体系。

#### FR-A01：指标定义与注册

定义 Agent 效果的量化指标体系，支持自定义指标注册。

- AC1：系统内置基础指标集，至少包括：任务完成率、任务耗时、用户满意度评分、重试率、错误率、输出质量评分。
- AC2：每个指标包含：ID、名称、计算公式、数据来源、采集频率、聚合方式（均值/中位数/百分位）。
- AC3：支持用户注册自定义指标，自定义指标与内置指标使用相同的存储和查询接口。
- AC4：指标支持分维度聚合——按 Agent、按任务类型、按时间窗口、按模型版本。

#### FR-A02：数据采集

从 Agent 运行时环境采集效果数据。

- AC1：采集器以事件驱动方式工作——Agent 完成任务时产生效果事件，采集器消费事件并写入度量存储。
- AC2：采集不阻塞 Agent 主任务执行，异步完成。
- AC3：效果事件至少包含：Agent ID、任务 ID、时间戳、指标值、上下文元数据（模型版本、Prompt 版本、知识库版本）。
- AC4：采集器支持宿主适配——不同宿主环境实现各自的事件生产者，AEO 定义统一的事件 schema。
- AC5：Change Event 采集与 Effect Event 对称——宿主环境在发生配置变更（模型升级、Prompt 修改、知识库更新、工具链变更）时产生 Change Event，AEO 定义 Change Event schema，宿主实现事件生产者。Change Event 至少包含：变更类型、变更内容摘要、变更前后版本标识、时间戳、影响的 Agent 范围。

#### FR-A03：基线管理

建立和维护效果基线，作为漂移检测的参照。

- AC1：基线是某个时间窗口内指标值的统计摘要（均值、标准差、百分位分布）。
- AC2：基线支持手动创建（用户指定时间窗口）和自动创建（系统在检测到稳定期时自动快照）。
- AC3：基线支持版本管理，新基线不覆盖旧基线，历史基线可追溯。
- AC4：基线绑定上下文标签（模型版本、Prompt 版本），支持按上下文筛选对应基线。

### 域 B：漂移检测（Drift Detection）

自动发现 Agent 效果偏离基线的异常。

#### FR-B01：统计漂移检测

基于统计方法检测指标值偏离基线的程度。

- AC1：支持滑动窗口检测——持续比较最近 N 个数据点与基线的偏离度。
- AC2：偏离度超过可配置阈值时生成漂移事件。阈值支持按指标独立配置。
- AC3：漂移事件包含：漂移指标、偏离方向（上升/下降）、偏离幅度、基线参照、触发时间、影响范围（哪些 Agent/任务类型）。
- AC4：支持多指标联合漂移检测——单个指标未超阈值但多个指标同时轻微偏移时，触发联合漂移告警。

#### FR-B02：变更关联检测

将效果变化与系统变更事件关联，识别变更引发的漂移。

- AC1：系统追踪变更事件：模型升级、Prompt 修改、知识库更新、工具链变更、配置调整。
- AC2：漂移事件发生时，自动回溯时间窗口内的变更事件，标注可能关联的变更。
- AC3：关联结果包含变更事件详情、时间相关性评分和影响范围重叠度。

#### FR-B03：告警与通知

漂移检测触发后，按配置的通知策略告警。

- AC1：告警通道支持扩展——AEO 定义告警事件 schema，宿主环境实现具体通道（IM、邮件、Webhook）。
- AC2：告警支持分级：warning（轻微偏移，持续观察）、critical（显著偏移，需要干预）。
- AC3：告警支持静默期配置，同一漂移事件在静默期内不重复告警。
- AC4：告警内容包含漂移摘要、影响评估和建议的下一步操作。

### 域 C：根因诊断（Root Cause Diagnosis）

分析效果漂移的原因，给出可验证的归因结论。

#### FR-C01：自动归因分析

漂移事件触发后，自动执行归因分析。

- AC1：归因分析检查以下维度：模型变更、Prompt 变更、知识库变更、输入分布变化、工具链变更、外部依赖变化。
- AC2：每个维度的归因结论包含：是否相关（是/否/不确定）、置信度、支撑证据（具体的变更记录、数据对比）。
- AC3：归因结论按置信度排序，最可能的根因排在最前。
- AC4：归因分析支持生成对比实验方案——给出实验设计（对比输入集、变更前后配置、预期观测指标），由宿主环境决定是否执行。实验结果由宿主回写后纳入归因证据。

#### FR-C02：诊断报告

将归因分析结果整理为结构化诊断报告。

- AC1：诊断报告包含：漂移概述、影响范围、根因排序、证据链、建议操作。
- AC2：报告支持按漂移事件 ID 检索。
- AC3：报告中的证据链可追溯到原始数据（效果事件、变更记录、对比实验结果）。

### 域 D：优化建议（Optimization Recommendation）

基于诊断结果生成可执行的优化方案。

#### FR-D01：优化方案生成

根据根因诊断结果，生成针对性的优化方案。

- AC1：优化方案类型至少包括：Prompt 调整建议、模型回退/切换建议、知识库补充建议、工具链配置调整建议。
- AC2：每个方案包含：操作步骤、预期效果、风险评估、回退方案。
- AC3：方案按预期收益和实施成本排序。
- AC4：方案与诊断报告关联，可追溯到对应的根因。

#### FR-D02：方案评审与生命周期管理

优化方案的评审、批准和状态追踪。AEO 负责方案的生命周期管理，实际执行由宿主环境完成。

- AC1：方案状态包括：proposed（已生成）、approved（已批准）、executing（宿主执行中）、completed（宿主已执行）、rejected（已拒绝）。
- AC2：高风险方案（涉及模型切换、大范围 Prompt 修改）需要人工批准后才能推荐给宿主执行。风险等级由方案的影响范围和回退复杂度决定。
- AC3：宿主回写方案执行完成状态后，自动触发闭环验证（FR-E01）。
- AC4：方案执行记录由宿主环境回写，AEO 负责持久化和关联。记录包含执行时间、执行者、实际操作和执行结果。宿主通过 `optimization-executed` 事件回写状态变更。

### 域 E：数据闭环（Data Closed Loop）

驱动"采集→分析→优化→验证"的完整循环。

#### FR-E01：闭环验证

优化方案执行后，自动验证效果是否改善。

- AC1：验证方式为基线对比——用优化前的基线和优化后的新数据对比，判断目标指标是否改善。
- AC2：验证结果包含：目标指标变化（绝对值和百分比）、是否达到预期、副作用检测（其他指标是否恶化）。
- AC3：验证未通过时，自动生成后续诊断任务，进入下一轮闭环。
- AC4：验证通过时，自动更新基线为优化后的水平。

#### FR-E02：闭环状态追踪

追踪每个漂移事件从发现到解决的完整生命周期。

- AC1：闭环状态包括：detected（已检测）、diagnosing（诊断中）、diagnosed（已诊断）、optimizing（优化中）、verifying（验证中）、resolved（已解决）、unresolved（未解决）。
- AC2：每个状态变迁记录时间戳和触发条件。
- AC3：超过可配置时间未推进的闭环自动升级告警级别。
- AC4：闭环历史可按 Agent、时间范围、根因类型检索，支持趋势分析。

#### FR-E03：效果报告

定期生成效果运营报告。

- AC1：报告周期可配置（日报/周报/月报）。
- AC2：报告内容包含：指标趋势、漂移事件统计、根因分布、优化成果、未解决问题。
- AC3：报告支持按 Agent、按任务类型、按时间范围筛选。
- AC4：报告可导出为 Markdown 格式。

### 域 F：Web 驾驶舱（Web Dashboard）

提供效果运营的可视化管理界面。

#### FR-F01：效果总览仪表盘

全局效果状态概览。

- AC1：展示核心 KPI 卡片：整体任务完成率、平均质量评分、活跃漂移事件数、未解决闭环数。
- AC2：KPI 卡片展示趋势（与上一周期对比的变化方向和幅度）。
- AC3：Agent 列表按效果健康度排序，异常 Agent 置顶。

#### FR-F02：漂移事件视图

漂移事件的列表和详情。

- AC1：列表支持按状态、严重级别、时间范围筛选。
- AC2：详情页展示漂移指标图表（时间序列 + 基线参照线）、关联变更、诊断报告、优化方案和验证结果。
- AC3：从漂移事件可一键跳转到对应的诊断报告和优化方案。

#### FR-F03：指标探索器

交互式指标查询和可视化。

- AC1：支持选择指标、时间范围、聚合维度，生成时间序列图表。
- AC2：支持多指标叠加对比。
- AC3：支持在图表上标注变更事件和漂移事件，直观展示因果关系。

## 4. 非功能需求（NFR）

### 4.1 性能

- NFR-01：效果事件采集到可查询的延迟 ≤ 60 秒（P95）。
- NFR-02：漂移检测从数据就绪到告警发出的延迟 ≤ 5 分钟。
- NFR-03：指标查询在 10 万条效果事件规模下响应时间 ≤ 3 秒（P95）。

### 4.2 可靠性

- NFR-04：效果事件写入后不丢失，支持持久化存储。
- NFR-05：漂移检测服务故障恢复后，自动补检故障期间的数据。
- NFR-06：闭环状态机保证状态一致性，不出现状态跳跃或丢失。

### 4.3 可扩展性

- NFR-07：指标类型可扩展，新增指标不修改已有采集和检测逻辑。
- NFR-08：归因维度可扩展，新增维度只需实现统一的归因分析接口。
- NFR-09：告警通道可扩展，新增通道只需实现统一的通知接口。

### 4.4 安全性

- NFR-10：效果数据支持访问控制，不同用户只能查看授权范围内的数据。
- NFR-11：诊断报告中不暴露敏感的系统内部实现细节。

### 4.5 NFR 验收标准

- AC-NFR-01：效果事件采集在 1000 条/分钟吞吐下满足延迟指标。
- AC-NFR-02：模拟漂移场景下，检测到告警的端到端延迟满足指标。
- AC-NFR-03：更换底层时序存储后，上层查询接口行为不变。

## 5. 概念架构

### 5.1 核心对象类型

- **Metric Definition**：指标定义，描述一个效果指标的计算方式和采集规则。包含 ID、名称、公式、数据来源、采集频率、聚合方式。
- **Effect Event**：效果事件，Agent 完成任务时产生的原始度量数据点。包含 Agent ID、任务 ID、时间戳、指标值集合、上下文元数据。
- **Baseline**：基线快照，某个时间窗口内指标值的统计摘要。绑定上下文标签，支持版本管理。
- **Drift Event**：漂移事件，检测到指标偏离基线时生成。包含漂移指标、偏离方向和幅度、基线参照、影响范围。
- **Change Event**：变更事件，系统配置或环境发生变化时记录。包含变更类型、变更内容、时间戳、影响范围。
- **Diagnosis Report**：诊断报告，归因分析的结构化产出。包含漂移概述、根因排序、证据链、建议操作。
- **Optimization Plan**：优化方案，基于诊断结论生成的可执行改进计划。包含操作步骤、预期效果、风险评估、回退方案。
- **Closed Loop**：闭环实例，追踪一个漂移事件从发现到解决的完整生命周期。绑定漂移事件、诊断报告、优化方案和验证结果。
- **Metric Store**：度量存储抽象接口，定义效果事件写入、指标时间序列查询和聚合计算的统一接口。与具体时序数据库解耦，宿主环境提供实现。核心操作：写入效果事件、按时间范围和维度查询指标序列、计算聚合统计（均值/百分位/标准差）。

### 5.2 闭环状态机

- **detected**：漂移已检测，等待诊断。
- **diagnosing**：归因分析执行中。
- **diagnosed**：诊断完成，等待优化方案。
- **optimizing**：优化方案执行中。
- **verifying**：闭环验证执行中。
- **resolved**：验证通过，漂移已修复。
- **unresolved**：验证未通过或超时未处理。

状态流转：

- detected → diagnosing：自动触发归因分析。
- diagnosing → diagnosed：归因分析完成。
- diagnosed → optimizing：优化方案被批准并开始执行。
- optimizing → verifying：方案执行完成，触发验证。
- verifying → resolved：验证通过。
- verifying → diagnosed：验证未通过，回到诊断阶段重新分析。
- 任意状态 → unresolved：超时未推进。

### 5.3 管线流转

效果度量管线：

```
Agent 完成任务
  → 产生 Effect Event（异步，不阻塞主任务）
  → 采集器消费事件，写入度量存储
  → 聚合计算，更新指标时间序列
```

漂移检测管线：

```
指标时间序列更新
  → 滑动窗口统计
  → 与 Baseline 对比
  → 偏离超阈值 → 生成 Drift Event
  → 关联 Change Event（时间窗口内的变更）
  → 触发告警通知
```

闭环管线：

```
Drift Event 生成
  → 创建 Closed Loop 实例（detected）
  → 自动归因分析（diagnosing → diagnosed）
  → 生成 Optimization Plan
  → 人工/自动批准 → 执行优化（optimizing）
  → 闭环验证（verifying）
  → 基线对比 → resolved / 回到 diagnosed
```

### 5.4 系统边界

- AEO 管理效果度量、漂移检测、根因诊断、优化建议和闭环验证。
- AEO 不负责 Agent 的任务执行和流程编排（那是 SEVO 的职责）。
- AEO 不负责知识的提取、存储和迭代（那是 KIVO 的职责）。
- AEO 不负责设计产物生成（那是 Claw Design 的职责）。
- AEO 的效果数据采集依赖宿主环境提供的事件接口，AEO 定义事件 schema，宿主负责生产事件。
- AEO 的优化建议中涉及知识库补充的部分，通过触发 KIVO 的调研任务实现，AEO 不直接操作知识库。

### 5.5 进化机制

- 检测回路：根据漂移检测的误报率和漏报率，调整检测阈值和窗口参数。
- 诊断回路：根据归因准确率反馈，优化归因分析的维度权重和证据评估策略。
- 建议回路：根据优化方案的采纳率和实际效果，优化方案生成策略。
- 基线回路：根据闭环验证结果，自动更新基线，适应系统的正常演进。

## 6. 与 Self-Evolving Harness 其他模块的边界

### 6.1 与 KIVO 的边界

KIVO 负责知识资产的治理。AEO 负责 Agent 效果的度量和优化。

- AEO 消费 KIVO 的数据：知识库版本变更作为 Change Event，用于变更关联检测。
- AEO 触发 KIVO 的能力：诊断发现知识缺失是根因时，触发 KIVO 的缺口检测和调研任务。
- 边界结论：AEO 管"做得怎么样"，KIVO 管"知道什么"。AEO 不直接修改知识库，KIVO 不直接度量效果。

### 6.2 与 SEVO 的边界

SEVO 负责研发流水线的阶段推进和交付闭环。AEO 负责交付后的效果持续监控。

- AEO 消费 SEVO 的数据：研发流程中的变更记录（Prompt 修改、代码部署）作为 Change Event。
- AEO 触发 SEVO 的能力：优化方案涉及代码修改时，可触发 SEVO 的研发流程。
- 边界结论：SEVO 管"怎么把改动做完"，AEO 管"改动之后效果怎么样"。

### 6.3 与宿主环境的边界

- 宿主环境负责：Agent 运行、效果事件生产、告警通道实现、优化方案的实际执行。
- AEO 负责：指标定义、漂移检测算法、归因分析逻辑、方案生成策略、闭环状态管理。
- 边界结论：AEO 定义效果运营的通用语义，宿主做运行时适配。

## 7. 约束与假设

### 8.1 约束

- AEO 定义通用效果运营语义，不预设某个唯一宿主。
- 效果事件的 schema 与具体 Agent 框架解耦，宿主实现适配层。
- 度量存储接口与具体时序数据库解耦。
- 优化方案的实际执行依赖宿主环境，AEO 只定义方案不直接执行。

### 8.2 假设

- 宿主环境能在 Agent 任务完成时产生效果事件。
- 宿主环境能提供系统变更事件（模型升级、配置修改等）。
- 用户接受 AEO 在后台持续采集和分析效果数据。
- 效果事件规模在产品初期控制在日均万级以内。

### 8.3 非目标

- AEO 不负责 Agent 的任务执行和流程编排（SEVO 的职责）。
- AEO 不负责知识的提取、存储和迭代（KIVO 的职责）。
- AEO 不负责设计产物生成（Claw Design 的职责）。
- AEO 不替代 APM 或基础设施监控，AEO 关注的是 Agent 业务效果，不是系统性能指标。
